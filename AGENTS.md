# AGENTS.md

Solana swap transaction parser library (Go).

## Project structure

- Library lives in `solanaswap-go/` package.
- `main.go` at root is a runnable example / manual test harness that fetches a live transaction from Solana mainnet and prints parsed swap data. Not a real CLI tool.
- Subpackages `meteora_dlmm_program/`, `meteora_damm_v2/`, `meteora_pools_program/` contain code **generated by `anchor-go`** from IDL JSON. Do not edit by hand.

## Build / verify

- `go build ./...` — builds library and `main.go` example.
- `go run main.go` — runs example against live RPC. Requires internet (the hardcoded `.vscode/launch.json` proxy `http://127.0.0.1:7897` may not exist; running directly usually works without it).
- `go test ./...` — **not meaningful**. The only `*_test.go` files are empty stubs generated by `anchor-go`.
- There is no CI, Makefile, linter config, or pre-commit setup.

## Adding a new Router (Aggregator / Trading Bot)

Most new routers are just wrappers that invoke real AMMs via inner instructions. For these, integration is **two lines**:

1. Add the program public-key constant to `solanaswap-go/consts.go`.
2. Append the constant to the `routerPrograms` slice in `solanaswap-go/parser.go`:

```go
var routerPrograms = []solana.PublicKey{
    // ... existing routers
    MY_NEW_ROUTER_PROGRAM_ID,
}
```

That's it. `ParseTransaction()` will automatically delegate to `processRouterSwaps()`, which scans inner instructions for known AMM calls (Raydium, Orca, Meteora, PumpFun AMM, etc.).

### ⚠️ Routers that need special handling

Some routers do NOT emit standard SPL transfers or AMM inner calls. They require bespoke parsers and must NOT be added to `routerPrograms`:

| Router | Program ID | Handler | Why special |
|--------|-----------|---------|-------------|
| Jupiter Aggregator | `JUP6LkbZbjS1jKKwapdHNy74zcZ3tLUZoi5QNyVTaV4` | `processJupiterSwaps()` | Event-based swap data |
| DFlow Aggregator v4 | `DF1ow4tspfHX9JwWJsAb9epbkA8hmpSEAtxXy1V27QBH` | `processJupiterSwaps()` | Shares Jupiter event format |
| OKX DEX Router | `6m2CDdhRgxpH4WjvdzxAYbGxwdGUz5MziiL5jek2kBma` | `processOKXSwaps()` | Custom event format |
| Moonshot | `MoonCVVNZFSYkqNXP6bxHLPL6QQJiMagDL3qcqUQTrG` | `processMoonshotSwaps()` | Custom event format |

If you add a router and get **empty swap results**, check the transaction logs. If the router emits custom events instead of SPL transfers / AMM calls, it needs a dedicated parser like the ones above.

## Adding support for a new AMM (Liquidity Program)

If the swap hits a new AMM program directly (not via a router), or the router delegates to an AMM we don't yet recognize:

1. Add program public-key constant to `solanaswap-go/consts.go`.
2. Add swap-type constant (`SwapType`) if needed.
3. Wire parsing in `solanaswap-go/parser.go`:
   - `ParseTransaction()` — add the program ID to the outer-instruction switch.
   - `processRouterSwaps()` — add the program ID to the inner-instruction switch (so routers can use it).
   - `setTxPoolInfo()` — add account-index mapping for pool / pool-in / pool-out accounts, plus discriminator whitelist.
4. If the AMM emits Anchor events, add event parsing in a dedicated `event_<name>.go` file (see `event_jupiter.go`, `event_pumpfun.go` for patterns).
5. If parsing relies on SPL `Transfer` / `TransferChecked` instructions, reuse helpers in `parse_transfer.go` / `parse_transfer_check.go`.

## PumpFun AMM discriminators

PumpFun AMM has multiple swap instruction variants. When adding new variants, update **all three locations**:

- `event_pumpswap.go` — add the 8-byte discriminator constant + `isXxxDiscriminator()` helper.
- `event_pumpfun.go` — add the discriminator to:
  - `processPumpfunAMMSwaps()` (inner instruction mode)
  - the outer-instruction switch
  - `setPumpFunSwapTxInfo()`

Current supported discriminators:

| Instruction | Discriminator Bytes | Go Literal |
|------------|---------------------|------------|
| `global:buy` | `102, 6, 61, 18, 1, 218, 235, 234` | `PumpFunAMMBuyDiscriminator` |
| `global:sell` | `51, 230, 133, 164, 1, 127, 131, 173` | `PumpFunAMMSellDiscriminator` |
| `global:buy_exact_quote_in` | `198, 46, 21, 82, 180, 217, 232, 112` | `PumpFunAMMBuyExactQuoteInDiscriminator` |
| `global:sell_exact_in` | `149, 39, 222, 155, 211, 124, 152, 26` | `PumpFunAMMSellExactInDiscriminator` |

To calculate a new discriminator:
```bash
go run -e 'package main; import ("crypto/sha256"; "encoding/hex"; "fmt"); func main() { h := sha256.Sum256([]byte("global:xxx")); fmt.Println(hex.EncodeToString(h[:8])) }'
```

## Mint decimals lookup

When building `TxInfo` for event-based AMMs, **always use `p.splDecimalsMap[mint.String()]`** to get token decimals. Do NOT use `p.splTokenInfoMap[accountAddr.String()].Decimals` — the latter maps token **account addresses** to mint info, not mint addresses, and will silently return 0 (or SOL's 9 decimals) for most tokens.

## Regenerating Anchor bindings

Commands from README (requires `anchor-go` and `anchor` CLI):

```bash
# Convert IDL format
anchor idl convert meteora_dlmm_program/idl.json -o meteora_dlmm_program/amm.json

# Generate Go bindings
anchor-go -idl meteora_dlmm_program/amm.json -output . -name meteora_dlmm_program
```

## API usage notes

- Two constructors:
  - `NewParser(tx *rpc.GetTransactionResult)` — from RPC `GetTransaction` result.
  - `NewTransactionParser(tx *solana.Transaction, txMeta *rpc.TransactionMeta)` — from raw transaction + meta.
- Typical flow: `ParseTransaction()` -> `ProcessSwapData()`.
- `ParseTransaction()` returns a slice of `SwapData` (one per leg); `ProcessSwapData()` collapses multi-leg swaps into a single `SwapInfo`.
- Jupiter and Pumpfun have special-cased logic in `ProcessSwapData` (event-based rather than transfer-based).

## Constraints / gotchas

- Custom program swaps are **not supported** if they bypass the outer-instruction check.
- `SwapInfo.Timestamp` is not populated from the transaction (must be fetched from the block separately).
- `splTokenInfoMap` and `splDecimalsMap` are built once per parser from `tx.Meta`; logic improvements are noted in README as TODO.
- Routers that only wrap AMM calls are handled generically via `routerPrograms` + `processRouterSwaps`.
